# TSK-023 — Nuova richiesta multi-step page

## Obiettivo

Implementare la pagina di creazione richiesta del dipendente (RF-M, F6/F7/F8): step 1
selezione tipo, step 2 form dinamico per tipo, step 3 riepilogo + invio. Integra i
form Zod da TSK-007.

## Componenti da creare

```
app/(employee)/requests/new/page.tsx        ← CSR: wizard richiesta
components/employee/requests/new/
  RequestTypeSelector.tsx                   ← step 1: 4 card tipo con icona e descrizione
  RequestFormAbsence.tsx                    ← step 2a: tipo assenza + date range
  RequestFormSwap.tsx                       ← step 2b: selezione turno collega + preview
  RequestFormNewShift.tsx                   ← step 2c: data + tipologia turno desiderata
  RequestFormModifyShift.tsx                ← step 2d: seleziona turno proprio + modifica proposta
  RequestReviewStep.tsx                     ← step 3: riepilogo + submit
  ColleagueTurnPicker.tsx                   ← componente ricerca turni colleghi (step 2b)
```

### Step 1 — Selezione tipo

Quattro card cliccabili con icona Lucide + titolo + breve descrizione:
- Assenza (ferie / malattia / permesso)
- Scambio turno (proposta a un collega)
- Nuovo turno (disponibilità a coprire)
- Modifica turno (modifica a un proprio turno)

### Step 2a — Assenza

```typescript
// Integra requestCreateSchema da lib/zod/ (TSK-007)
// Campi: tipoAssenza (radio), dataInizio (date picker), dataFine (date picker), note
// Validazione: fine >= inizio
// Anteprima: "Richieda X giorni lavorativi"
```

### Step 2b — Scambio turno

```typescript
// ColleagueTurnPicker:
//   GET /api/shifts?available_for_swap=true  (solo turni di colleghi della stessa fascia)
//   Lista filtrata per: data, tipologia, qualifica collega
// Mostra: il turno del dipendente da cedere (auto: selezionabile tra i propri turni)
// Preview impatto: RB-10 client-side via lib/rules (ottimistico)
// RF-M nota: il collega deve accettare prima dell'approvazione admin (F7)
```

### Step 2c — Nuovo turno

- Data singola (date picker) + tipologia desiderata (select)
- Note facoltative

### Step 2d — Modifica turno

- Selezione turno proprio (from `GET /api/shifts?mine=true&future=true`)
- Proposta di modifica: nuovo orario start/end o nuova tipologia
- Preview impatto client-side (RB-01..08 ottimistico)

### Step 3 — Riepilogo + invio

```typescript
// Riepilogo leggibile di tutto il payload
// POST /api/requests → onSuccess: toast + redirect a /requests
// onError 400: mapping Zod issues ai campi → torna step 2 con errori
```

### Sicurezza (T-SEC-04, RB-13)

Il payload inviato include solo i campi consentiti per il dipendente (schema da TSK-007 `requestCreateSchema`). Il BE riapplica RB-13 prima di salvare.

## Criteri di accettazione

- [ ] Step 1 → step 2: selezione tipo mostra form corretto; cambio tipo possibile con "Indietro"
- [ ] Step 2a (Assenza): date fine < inizio → errore inline; submit → richiesta creata, redirect
- [ ] Step 2b (Scambio): ColleagueTurnPicker mostra solo turni dello stesso orario/qualifica
- [ ] Step 2b: preview RB-10 ottimistico mostra eventuali conflitti prima dell'invio
- [ ] Step 3: "Conferma e invia" disabilitato durante il POST; spinner visibile
- [ ] Post-invio: redirect a `/requests`, nuova richiesta in lista con stato `sent`
- [ ] Admin riceve notifica SSE `request.received` entro 10s (T-INT-03)
