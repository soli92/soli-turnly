# TSK-026 — Swap admin page

## Obiettivo

Implementare la schermata admin "Scambio turni" (RF-F, screen 7 inventario) e completare
la logica di `POST /api/admin/swap` (attualmente stub 501 con TODO TSK-006).

Il flusso: admin seleziona due turni di dipendenti diversi → anteprima impatto (RB-10 —
rivalidazione per entrambe le parti) → se solo avvisi: conferma esplicita → swap eseguito.

Il modulo `validateSwap` è già presente in `lib/rules/validateSwap.ts`.
La tabella `swap_operations` esiste già (TSK-013).

## Componenti / Endpoint da creare

### Pagina (fe)

```
app/(admin)/swap/
  page.tsx                         ← RSC: header + verifica ruolo admin
  _components/
    SwapAdminPageClient.tsx         ← Client: orchestrator state machine (selezione → anteprima → conferma)
    ShiftSearchPanel.tsx            ← ricerca turni per dipendente + data (autocomplete utenti, date picker)
    SelectedShiftCard.tsx           ← scheda turno selezionato (dipendente, orari, tipologia)
    SwapImpactPreview.tsx           ← pannello anteprima: RB-10 outcome per entrambe le parti
    SwapViolationSummary.tsx        ← lista violazioni bloccanti / avvisi (colori rosso/giallo)
    SwapConfirmDialog.tsx           ← dialog di conferma quando ci sono solo avvisi (RF-F CA2)
```

### Implementazione API (be)

`app/api/admin/swap/route.ts` — sostituire stub 501 con logica completa:

1. Recupera `shiftA` e `shiftB` dal DB
2. Verifica che appartengano a dipendenti diversi (`shiftA.userId !== shiftB.userId`)
3. Esegue `validateSwap(shiftA, shiftB, allShiftsUserA, allShiftsUserB)` via `lib/rules/validateSwap.ts`
4. Se outcome ha blocking violations → risponde 422 con dettaglio (nessun cambio DB)
5. Se solo warnings → accetta con `?confirm=true`; senza flag → risponde 200 con `{ requiresConfirmation: true, warnings: [...] }`
6. Su commit: `UPDATE shifts SET userId = ? WHERE id = ?` su entrambi; inserisce riga `swap_operations`; logga su `audit_log`

```typescript
// Risposta 422 — violazione bloccante
{ outcome: 'rejected', blocking: [{ party: 'A'|'B', rule: 'RB-01', message: '...' }] }

// Risposta 200 — solo avvisi, richiede conferma
{ outcome: 'warnings', requiresConfirmation: true, warnings: [{ party: 'A'|'B', rule: 'RB-02', message: '...' }] }

// Risposta 200 — swap eseguito
{ outcome: 'executed', swapOperationId: '<uuid>' }
```

### Endpoint anteprima (be)

```
GET /api/admin/swap/preview?shiftAId=<uuid>&shiftBId=<uuid>
```

Restituisce l'esito RB-10 senza modificare il DB. Usato da `SwapImpactPreview` per mostrare
l'anteprima prima che l'admin confirmi.

## Acceptance Criteria

- [ ] Swap valido (nessuna violazione): lo swap viene eseguito, i `userId` sui due turni sono scambiati nel DB, `swap_operations` ha una riga con `origin='admin'`
- [ ] RF-F CA1: swap che genera violazione bloccante RB-01 per una delle parti → risposta 422, DB invariato, UI mostra quale parte e quale regola
- [ ] RF-F CA2: swap con solo avvisi → UI mostra `SwapConfirmDialog`; admin può procedere con conferma o annullare
- [ ] RF-F CA4: ogni swap confermato produce una riga in `audit_log` con `action='swap.admin'`, `before`, `after`
- [ ] `POST /api/admin/swap` da dipendente (non-admin) → 403
- [ ] Selezione dello stesso turno per A e B → errore validazione client "i turni devono appartenere a dipendenti diversi"
- [ ] `SwapImpactPreview`: carica automaticamente quando entrambi i turni sono selezionati (nessun click extra)
- [ ] Stato iniziale: i due pannelli di selezione mostrano placeholder "Seleziona turno"
- [ ] Layout mobile: pannelli A e B in colonna su 375px, anteprima sotto
