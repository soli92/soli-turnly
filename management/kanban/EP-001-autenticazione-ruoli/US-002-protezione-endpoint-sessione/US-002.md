# US-002 — Protezione endpoint per ruolo e autorizzazione server-side

## Descrizione
Come sistema, voglio verificare il ruolo dell'utente su ogni richiesta server-side,
affinché nessun utente possa accedere a risorse o operazioni al di fuori del proprio ruolo,
anche aggirando la UI.

## Business Rules
- Un dipendente che accede a un URL o endpoint riservato all'admin riceve 403, non un redirect cosmetico (RF-A CA2).
- La protezione IDOR impedisce l'accesso a risorse altrui manipolando identificativi in URL, parametri o payload.
- Ogni permesso è verificato lato server su ogni richiesta; nascondere un controllo nella sola UI non è sufficiente (RNF-Sicurezza).
- L'autorizzazione a livello di campo sul profilo rifiuta i campi contrattuali se modificati dal dipendente (RB-13).
- Le operazioni di scrittura rilevanti (assegnazioni, swap, assenze, modifiche anagrafiche, decisioni su richieste) sono tracciate nell'AuditLog.

## UI Reference
Nessuna schermata dedicata; si manifesta come risposta 403/401 a chiamate non autorizzate.

## Acceptance Criteria
- [ ] Un dipendente che chiama un endpoint di pianificazione o anagrafica admin riceve 403 (T-SEC-02).
- [ ] Un dipendente che richiede i turni passando l'id di un collega (IDOR) riceve solo i propri o un 403, mai i turni altrui (T-SEC-01).
- [ ] Accesso a risorsa per id inesistente o non autorizzato: 404/403 coerente, nessuna fuga d'informazione (T-SEC-03).
- [ ] Un dipendente che tenta di modificare qualifica/ore contrattuali/ruolo via API riceve 403; il valore non cambia (T-SEC-05, RB-13).
- [ ] Un dipendente che tenta di approvare una richiesta (propria o altrui) via API riceve 403 (T-SEC-06).
- [ ] Tutte le operazioni di scrittura rilevanti generano una voce nell'AuditLog con attore, azione e snapshot prima/dopo.
