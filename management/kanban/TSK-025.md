# TSK-025 — Disponibilità e preferenze dipendente

## Obiettivo

Implementare la schermata "Disponibilità e preferenze" del dipendente (RF-L, screen 17
inventario) e il relativo layer API. Il dipendente inserisce/modifica/elimina finestre di
disponibilità (available/unavailable/preference) ricorrenti o su date specifiche.

La pagina profilo (`app/(employee)/profile/`) è già completa (ProfileForm + PasswordForm).
Questo task aggiunge la pagina dedicata alla disponibilità e implementa i tre endpoint
attualmente stub 501 in `app/api/users/me/availability/route.ts`.

## Componenti / Endpoint da creare

### Pagina (fe)

```
app/(employee)/availability/
  page.tsx                        ← RSC: fetch lista disponibilità via API
  _components/
    AvailabilityPageClient.tsx    ← Client: TanStack Query, stato lista
    AvailabilityList.tsx          ← lista card con tipo/scope/definizione + azioni
    AvailabilityForm.tsx          ← form crea/modifica: tipo, scope, definizione (jsonb)
    AvailabilityCard.tsx          ← singola voce con badge tipo (verde/rosso/giallo)
```

### Schema Zod condiviso (be+fe)

```typescript
// lib/zod.ts — aggiungere:
export const availabilityCreateSchema = z.object({
  type: z.enum(['available', 'unavailable', 'preference']),
  scope: z.enum(['recurring', 'date_range']),
  definition: z.union([
    // recurring
    z.object({ dayOfWeek: z.number().min(0).max(6), startTime: z.string(), endTime: z.string() }),
    // date_range
    z.object({ startDate: z.string(), endDate: z.string(), startTime: z.string().optional(), endTime: z.string().optional() }),
  ]),
  notes: z.string().max(500).optional(),
});
```

### API (be)

```
app/api/users/me/availability/route.ts   ← implementa GET / POST / DELETE
```

- `GET /api/users/me/availability` — lista disponibilità dell'utente autenticato, ordinata per `createdAt DESC`
- `POST /api/users/me/availability` — crea una voce; valida con `availabilityCreateSchema`
- `DELETE /api/users/me/availability?id=<uuid>` — elimina voce; verifica ownership (`userId = session.user.id`)

### Sicurezza (RB-13, T-SEC-05)

- L'ID utente viene sempre dal token JWT (`session.user.id`), mai da query params
- Il `DELETE` verifica `WHERE id = ? AND userId = session.user.id` prima di eliminare
- Un dipendente non può leggere o modificare disponibilità altrui → 403 se `userId` non corrisponde

## Acceptance Criteria

- [ ] `GET /api/users/me/availability` restituisce array vuoto `[]` per nuovo utente e lista ordinata se ci sono voci
- [ ] `POST /api/users/me/availability` con payload valido `scope=recurring` crea riga in tabella `availability`; risponde 201 con il record creato
- [ ] `POST` con `definition.dayOfWeek=8` fallisce con 400 (Zod: dayOfWeek max 6)
- [ ] `DELETE /api/users/me/availability?id=<altrui>` risponde 403 — un dipendente non può eliminare disponibilità di un altro (RB-13, T-SEC-05)
- [ ] AvailabilityList: lista mostra tutte le voci con badge tipo (verde=available, rosso=unavailable, giallo=preference) e scope
- [ ] AvailabilityForm: submit con scope=date_range e campo `endDate < startDate` → errore di validazione inline
- [ ] Eliminazione voce: dopo `DELETE` la lista si aggiorna (invalidazione query TanStack)
- [ ] RF-L CA2: un turno admin che contraddice un'indisponibilità dichiarata genera avviso — questo CA viene verificato lato matrice turni (già coperto da RB-15 in TSK-006/TSK-007); la disponibilità inserita deve essere recuperabile dalla query della matrice
- [ ] Layout mobile 375px: form e lista usabili; nessun overflow orizzontale
- [ ] WCAG 2.2 AA: tutti i form con label associata, errori con `role="alert"`
